# 실행 안내 & 화면 연결 정리

내가 만든 **작업자화면(worker_v5)·대시보드(dashboard_v2_2)를 이 DB+백엔드에 실제로 연결**한 상태입니다.
입력화면에서 주문을 넣으면 → DB가 바뀌고 → 두 화면이 자동으로 갱신됩니다. PostgreSQL로 전체 흐름 검증 완료.

```
screen_system/
├─ schema.sql          DB 구조 (진행상태 0~6, 3공정 + 재고 테이블)
├─ main.py             ★ 진입점 — uvicorn main:app
├─ app.py              호환 shim (예전 uvicorn app:app 도 동작)
├─ config.py           설정·상수      schemas.py  요청 모델
├─ db.py               DB 접속        security.py 관리자 인증
├─ services/           비즈니스 로직 (stage / inventory / shipping /
│                      formatting / printing / mailer / scanner)
├─ routers/            HTTP 라우트 (pages, public, admin/*)
├─ check_api.py        API 회귀 검사 도구
├─ requirements.txt
└─ frontend/
   ├─ input.html       주문 전체(고객+명세) 수기 입력  ← 내가 손으로 입력
   ├─ worker.html      = 올려준 worker_v5 (USE_MOCK=false 로 백엔드 연결)
   └─ dashboard.html   = 올려준 dashboard_v2_2 (USE_MOCK=false 로 백엔드 연결)
```

## 0. 현장 PC 에서 최신 코드 받기 (pull 배포 순서)

> **사무실에서 먼저 할 것** — `git push origin main`
> 이걸 안 하면 현장에서 받을 게 없습니다.

### ① 현장 설정 파일 백업 (제일 먼저)

`printer_config.json` 과 `scanner_config.json` 은 **PC마다 값이 다른데 git 에 들어 있습니다.**
프린터 이름과 COM 포트가 노트북과 현장이 다르면 pull 이 막히거나 덮어써집니다.

```bat
cd 저장소폴더\screen_system
copy printer_config.json printer_config.json.bak
copy scanner_config.json scanner_config.json.bak
```

### ② pull

```bat
cd 저장소폴더
git status
git checkout main
git pull
```

> `git status` 에 위 두 파일이 나오면 → `git checkout -- screen_system\printer_config.json screen_system\scanner_config.json` 로 버린 뒤 pull 하고, ③에서 되돌립니다.
> 다른 파일이 나오면 **현장에서만 한 수정**이니 지우지 말고 먼저 확인하세요.

### ③ 설정 파일 되돌리기

pull 후 두 파일을 열어 **현장 값(프린터 이름·COM 포트)** 이 맞는지 확인하고,
바뀌었으면 `.bak` 내용으로 되돌립니다. 단 `printer_config.json` 에는 이번에 키가 하나 늘었으니 이 줄은 남겨두세요:

```json
"order_printer_small": "",
```

### ⑥ 프린터 용지 길이 변경 ★잊기 쉬움

라벨이 **449.2mm → 412mm** 로 바뀌었습니다.
주문라벨 프린터(Xprinter XP-TT426B)의 인쇄 기본 설정에서 용지 길이를 **412mm** 로 바꾸세요.
안 바꾸면 라벨이 잘리거나 빈 공간이 생깁니다.

**주의할 점 3가지**

1. **`screen_system` 폴더 안에서 실행해야 합니다.** 상위 폴더에서 `uvicorn screen_system.main:app` 으로 띄우면 import 가 깨집니다. (분리 전에도 마찬가지였습니다)

## 1. 처음 한 번: 설치
```bash
createdb screen
psql -d screen -f schema.sql
pip install -r requirements.txt
```

## 2. 매번: 실행
```bash
# Windows
set DATABASE_URL=postgresql://postgres:비번@localhost:5432/screen
# Mac/Linux/라즈베리파이
export DATABASE_URL=postgresql://postgres:비번@localhost:5432/screen

uvicorn main:app --host 0.0.0.0 --port 8000
```

> 기존 `uvicorn app:app` 도 그대로 동작합니다(app.py 가 main 을 재수출하는 shim).
> 새 스크립트는 `main:app` 을 쓰세요.

## 3. 브라우저 주소
| 화면 | 주소 |
|---|---|
| 입력 | http://localhost:8000/input |
| 작업자화면 | http://localhost:8000/worker |
| 대시보드 | http://localhost:8000/dashboard |

태블릿·대형모니터에서는 `localhost` 대신 PC의 IP로 접속 (예 `http://192.168.0.10:8000/worker`).

## 4. 동작 흐름
1. 입력화면에서 주문 저장 → 저장 시 **바코드(=製品番号)** 가 자동 생성됨 (`ORD-날짜-순번-01`).
2. 대시보드에 그 제품이 바로 나타남(3초 내). 라벨을 인쇄해 붙임.
3. 작업자가 라벨 바코드를 스캔 → 스캔 1번에 stage +1.
   - 裁断 시작(1)→완료(2)→ミシン 시작(3)→완료(4)→ハトメ 시작(5)→완료(6).
   - ハトメ完了(6)가 최종. 주문의 전 명세가 6이 되면 자동으로 出荷済 처리.
   - 잘못 스캔하면 화면의 「戻る」 = stage −1 (UNDO).
4. 대시보드의 상태·실적·이력이 자동 갱신됨.

## 5. 진행상태 모델 (핵심)
`order_items.current_stage` **숫자 하나(0~6)** 가 진행상태의 유일 기준.
홀수=작업중, 짝수=완료: `0 접수 / 1 재단중 2 재단완료 / 3 미싱중 4 미싱완료 / 5 하토메중 6 하토메완료`.
(梱包 7·8 은 폐지됨 — migration_remove_packing.sql)
모든 스캔은 `scan_events` 테이블에 이력으로 남음(대시보드 이력·현황 근거).

## 6. 백엔드 API (화면들이 부르는 주소)
작업자화면:
- `GET  /api/orders/{바코드}` → `{order_no, fabric, size, magictape, skirt, e_top, e_bottom, e_left, e_right, stage}`
- `POST /api/scan       {order_no}` → stage +1
- `POST /api/scan/undo  {order_no}` → stage −1

대시보드:
- `GET /api/board`       → 제품별 [{order_no, fabric, size, qty, opts{top,bottom,left,right}, received_at, status{cutting,sewing,eyelet,packing}}]
- `GET /api/events`      → 최근 스캔 이력 [{t, order_no, ev, text}]
- `GET /api/inventory`   → 생지 재고 [{fabric, remain, cap}]
- `GET /api/accessories` → 부속품 재고 [{name, remain, cap, unit}]

입력화면:
- `POST /api/orders {channel, customer_name, ..., items:[{product_type, fabric_type, width_mm, height_mm, quantity, process_top/bottom/left/right(+_mm), fire_cert_no}]}`
  → `{order_no, barcodes:[...]}`

## 7. 화면-DB 매핑 메모
- 화면의 **order_no(製品番号) = order_items.barcode** (라벨 1장 = 제품 1점 = 스캔 단위).
- 4변 가공 `process_top/bottom/left/right` 는 **하토메 전용**(none/eyelet)+mm.
  벨크로·스커트는 같은 변에 동시에 붙으므로 별도 컬럼으로 관리:
  - `velcro_sides` : NULL(없음) / 3(上左右) / 4(四辺)
  - `has_skirt`    : 스커트 유무(下辺). 취부방법은 `skirt_attachment`
- 재고는 QR 방식(`inv_item`/`inv_unit`/`inv_tx`). 관리자 「在庫」 탭에서 관리.
  그룹 7 消耗品 = 프린터 라벨지/리본/A4 등 공장 소모품.
